# CX-1000 WebAPI(HTTP)

CX-1000 內建一支官方 HTTP WebAPI，讓你的程式能以「網路呼叫」的方式，模擬按下某台機器的控制輸入 —— 等於一條軟體版的乾接點。

## 這是什麼

!!! note "一句話"
    WebAPI 是 CX-1000 的對內 HTTP 控制介面。你的軟體（中控平台、腳本、伺服器…）對 **SM1000** 發一個帶認證的 HTTP 請求，就能觸發系統內某台機器的「控制輸入」，進而啟動已綁定的廣播或緊急動作。

在此版本（v1.0）中，WebAPI **只開放一個功能**：模擬觸發控制輸入（`control-inputs-digest`）。雖然只有一項，但因為 CX 幾乎所有的廣播與緊急動作都能綁到「控制輸入」事件上，所以這一招足以用網路方式驅動大部分廣播行為。

## 能做什麼：軟體版乾接點

過去要讓外部設備觸發 CX 廣播，得走實體乾接點：`外部接點 → 繼電器 → 機器控制輸入`。WebAPI 讓你把最後這段換成一個 HTTP 呼叫。

```text
你的程式 ──HTTP GET(Digest 認證)──▶ CX-SM1000 :80
   └─ 模擬「某台機器的控制輸入 N」ON
        └─ CX 觸發該控制輸入綁定的事件(啟動廣播樣式 / 緊急訊息樣式…)
             └─ 廣播發送
```

!!! tip "整合思路"
    攝影機、人數統計 App 等設備先把事件送給中控軟體（webhook 或輪詢），中控軟體再對 SM1000 發一個 GET，即可觸發 CX 廣播 —— 免繼電器、免拉線。適合作為**軟體驅動**的觸發源。

!!! tetplus "分清楚兩件事：WebAPI（原廠）vs 中控軟體（可自研）"
    - **WebAPI 本身**屬 CX-1000 **原廠設計**的介面，但**公開零售韌體不一定內建**（見文末說明），屬「⚙️ 原廠但有門檻」。
    - 呼叫這支 API 的**中控軟體**則是「軟體驅動」的那一層：可用 **tet-tw 台灣分公司自行研發的中控軟體**（**非 TOA 原廠標準、非原廠背書**），或你自備的等效程式。

## 基本規格

| 項目 | 內容 |
|---|---|
| 協定 | HTTP/1.1 |
| 認證 | Digest 認證（使用系統的 ID/密碼；出廠預設 `cx-1000` / `guest`） |
| 編碼 | UTF-8 |
| 對象 | CX-SM1000 的 IP 位址 |
| 連接埠 | 預設 **80**（可於機器檢出工具 / Web 維護畫面變更） |
| 回應格式 | `application/json` |

!!! warning "請更換預設帳密"
    出廠帳密 `cx-1000` / `guest` 為公開資訊。既然這支 API 能在 LAN 上直接觸發廣播，正式上線前務必於系統設定變更登錄帳密，避免同網段任何人都能按下你的控制輸入。

## URL 格式

```text
http://<SM1000_IP>/api/v1/<category>/<command>?param1=值&param2=值…
```

- `<category>`：目前僅有 `system`（系統管理／機器控制）
- 子類別：`units`（對系統內某台已登錄機器做個別控制）
- 參數順序任意；`?` 分隔指令與參數、`&` 分隔各參數、`=` 分隔名稱與值

## 指令：`control-inputs-digest`

模擬觸發控制輸入。此指令需 **SM1000 韌體 v1.0.2 以上**。

**端點**

```text
GET /api/v1/system/units/{unit_id}/control-inputs-digest/{number}?is_pulse={t/f}&is_enabled={t/f}
```

| 參數 | 意義 | 必填 |
|---|---|---|
| `{unit_id}` | 機器 ID（整數） | 是 |
| `{number}` | 控制輸入編號（整數） | 是 |
| `is_pulse` | `true` = Pulse（約 1 秒後自動 OFF）／`false` = Level（維持） | 是 |
| `is_enabled` | `true` = ON／`false` = OFF（省略時預設 `true`） | 否 |

**成功回應**

```json
{"result":"OK"}
```

**範例（curl）—— 將「機器 ID 2 的控制輸入 1」以 Level 設為 ON**

```bash
curl --digest -u "cx-1000:guest" -X GET -H "Accept: application/json" \
  "http://<SM1000_IP>/api/v1/system/units/2/control-inputs-digest/1?is_pulse=false&is_enabled=true"
```

- `--digest -u "ID:密碼"`：填入系統登錄的帳密
- 使用 **GET**；實際 URL 為一整行

## 錯誤碼

| 狀態碼 | 意義 |
|---|---|
| `200 OK` | 成功 |
| `400 Bad Request` | 參數不足或不正確 |
| `401 Unauthorized` | 帳密錯誤／Digest 認證失敗 |

## 實務使用步驟

1. **CX 設定軟體**：選一台有控制輸入的機器（CC1016 控制 I/O 端子最多，站台或 AF1062 亦可），把某個控制輸入編號綁到你要的事件（例：General broadcast pattern start、Emergency message pattern start…）。
2. 記下該機器的 **機器 ID（unit_id）** 與 **控制輸入編號**。
3. 你的中控軟體（或 LAN 上任何程式）在事件發生時，對 **SM1000 的 IP** 發送該 GET（附 Digest 帳密）。
4. CX 就像收到實體乾接點一樣，觸發對應動作。

!!! note "WebAPI 只負責「按下」"
    WebAPI 傳達的仍是「控制輸入」語意，本身不含廣播內容。動作要播什麼、送到哪，取決於你事先在 CX 設定裡把該控制輸入綁定的事件。API 不會、也無法直接指定廣播目的地或音源。

## 限制與注意（v1.0）

!!! warning "功能與環境限制"
    - **目前只有這一項功能**（模擬控制輸入）。沒有直接下廣播指令、沒有狀態查詢、沒有音訊傳輸。
    - 需 **SM1000 韌體 v1.0.2 以上**。
    - 僅適用於 **LAN 內**、對 **SM1000 的 IP**、**port 80**。
    - 這支 WebAPI（port 80，inbound HTTP 控制）與手冊中的「Remote control 埠 15080」是**不同的東西**（15080 為系統對外送出的 remote protocol）。

!!! danger "韌體支援：不保證每台機器都有"
    這支 WebAPI 並非在所有出廠韌體上都預先開啟。**公開零售版韌體有可能未內建此 API**，需向 TOA 確認 / 申請對應（或特殊）韌體版本。上線前請務必以實機驗證。

    **快速判斷方法**：直接對該路由發請求，若回應為 **302 導向到登入頁**（而非 JSON 或 401），代表**該 API 路由在這台機器上並不存在**，即此韌體未提供 WebAPI，需先處理韌體。

!!! note "攝影機能否直接呼叫？"
    攝影機等設備能否**不透過中控軟體、自行直接**打這支 API，取決於該設備的 HTTP 通知功能是否支援 **Digest 認證**並能帶正確 URL。此點需**台架實測**，不可預設可行。

## 重點整理

- WebAPI 是 CX-1000 對內的 HTTP 控制介面，目標對象為 **SM1000**，預設 **port 80**、**Digest 認證**、回應 JSON。
- v1.0 唯一功能 `control-inputs-digest` = **以網路模擬觸發控制輸入**，等於軟體版乾接點；可用來由軟體驅動廣播／緊急動作。
- 使用前需先在 CX 設定把「目標機器的控制輸入編號」綁定到你要的事件；API 只負責「按下」。
- 需 **SM1000 韌體 v1.0.2 以上**，且**公開零售韌體不一定內建**此 API —— 若實測回應為 **302 導向登入頁，代表該路由不存在**，須向 TOA 確認韌體。
- 僅限 LAN 使用，務必更換出廠預設帳密；與對外的 remote control 埠 15080 是不同機制。
