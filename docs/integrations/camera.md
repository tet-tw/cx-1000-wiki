# 攝影機事件觸發廣播整合（TRIFORA N-C6 → IP-A1／CX-1000）

TRIFORA N-C6 系列網路攝影機偵測到畫面事件後，可用「HTTP 通知（純網路）」或「乾接點（硬體）」兩種方式，觸發指定的 IP-A1 喇叭或 CX-1000 廣播；其中純網路路線佈線最省、且能與 CX 廣播在同一顆喇叭上並存，靠優先權仲裁決定誰蓋誰。

!!! danger "先講清楚：這不是消防系統"
    本整合為一般保全／導引應用，**非 EN54／CNS 消防認證，也非 TOA 原廠背書的緊急疏散系統**。凡法規要求的火警偵測與疏散廣播，仍須採用合格的消防設備，不可由本方案取代。

## ✅ 原廠標準：在 OP 監看 IP 攝影機（External camera monitoring）

!!! note "這段是「原廠功能」，跟下方「事件觸發廣播」是兩回事"
    - **在 OP 監看攝影機** = CX-1000 **原廠標準功能**：把 ONVIF 攝影機影像顯示在 **OP1700／RM1700** 觸控螢幕，可邊看邊廣播，屬原廠手冊範圍。
    - **攝影機事件 → 觸發廣播**（本頁下半部）= 需接點或軟體，另一回事。

### 設定流程（TRIFORA N-C62550R3 實測成功）

**攝影機端（N-C6 網頁）**

1. **IP**：設不衝突的位址（**別用預設 `192.168.14.1`**，會撞 CX-SM1000）。
2. **認證**：`Basic → Security` → Authentication = **Basic**。
3. **ONVIF**（`Network → ONVIF`）：Available Codec 要**含 H.264**。
4. **串流**（`Video & Audio → Stream`）：把 CX 要取的那條（通常 **Stream 1**）設 **H.264／720p／15fps**（**不可 H.265**，CX 解不了）→ 按 **Set** 存檔。

**CX 端（設定軟體）**

1. `Device registration`：登錄這支 IP camera（名稱 + 攝影機 IP）。
2. `Network (Device) → IP camera`：填 **IP camera ID + Password**（= 攝影機登入帳密，例 `admin`）。
3. `Network (Device) → Video → IP camera receiving`：codec **H.264**、解析度 **720p**、**Frame rate 15**、I-Frame **15**。
4. `Event activation → Key`：某台 **OP1700 的 Direct key = External camera monitoring** →（1）IP camera **選這支攝影機**；要邊看邊廣播可再設 Broadcast destination。
5. **`Save` → 上傳**（`Communication → Settings & audio file upload`）。

**OP 操作**

- OP1700 觸控螢幕**點該 Direct key** → 攝影機影像**全螢幕**顯示。
- 邊看邊廣播：按機身**實體 Broadcast 鍵**；返回：按機身**實體 Cancel 鍵**。

### ⚠ 疑難排解（實測踩過的雷）

| 症狀 | 原因 | 解法 |
|---|---|---|
| 沒畫面 | 串流是 **H.265** | CX 只解 **H.264／JPEG** → 把該串流改 **H.264** 並按 Set |
| 沒畫面 | CX 要的 **fps 攝影機做不到**（如 20fps） | **CX 是主導方**，會把設定經 ONVIF 推到攝影機 → CX 的值**必須是攝影機支援的值**（N-C6 fps 只有 30/15/10/…**沒有 20** → 用 **15**） |
| I-frame 對不上 | 單位不同 | 攝影機＝**秒**、CX＝**幀**；**幀 = fps × 秒**（15fps → 1 秒 = 15 幀） |
| 連不上 | 認證 | 攝影機預設 Digest；CX 走 ONVIF 需要時改 **Basic** |
| 什麼都連不上 | **IP 撞** SM1000 的 `.1` | 改攝影機 IP |
| 分不清攝影機端 vs CX 端 | — | 同網段 PC 用 **VLC** 開 `rtsp://<攝影機IP>/unicast/codec1`（輸入帳密）；**放得出來 = 攝影機／帳密／網路 OK，問題在 CX 端**（多半 codec／fps／認證） |
| 設定都對還是沒反應 | 沒生效 | 一定要 **Save + 上傳到 SM1000** |

!!! tip "兩邊設定要一致，且用攝影機做得到的值"
    CX「IP camera receiving」會透過 ONVIF 把 codec／fps／解析度推到攝影機那條串流。**兩邊設成完全一樣、且是攝影機支援的值**（H.264／720p／15fps）最保險。

## 第一步：先看清楚攝影機「偵測」到什麼

這一步最容易誤會，務必先確認需求對得上攝影機能力。

!!! warning "N-C6 內建偵測是「動体偵測」，不是 AI 人物／滯留偵測"
    N-C6 系列（如 N-C62550R3）**內建**的偵測只有兩種，且互斥：

    - **動体偵測**：偵測畫面內的「動作變化」，**不分辨是不是人**。貓、車燈、樹影、雨雪、光影變化都可能誤觸。
    - **妨害偵測**：鏡頭被遮蓋或轉向（整個畫面大幅變化）。

    若客戶要的是「有人滯留、越線、闖入」這類**真正的 AI 人物辨識**，攝影機需**另外安裝擴充 App**（最多 3 支、其中 AI 類僅 1 支，安裝／移除需重開機），裝好後才會出現「擴充 App 事件」這個觸發源。其實際辨識能力請另向 TOA／代理確認，屬另議項目。

!!! tip "建議"
    先用動体偵測搭配「檢知對象大小、感度、反向遮罩（8×8 格）、排程」把誤觸壓低；需要精準人物／滯留判斷再評估 AI 擴充 App。

## 攝影機端：HTTP 通知怎麼設

攝影機的 HTTP(S) 通知是**網路側內建功能**，不需額外的接點選配線、也不需 PC。可自訂度很高，能組出接收端要的任何請求。

設定路徑：

```
攝影機網頁 → 程式事件（プログラムイベント）
   事件 = 動体偵測 → 動作 = HTTP 通知 1～20

攝影機網頁 → 通知先設定 → HTTP(S) 通知
   填入 URL、Method、認證、通知內容
```

主要可設項目：

| 項目 | 內容／上限 |
|---|---|
| 通知先數量 | 最多 20 個（可各自指定不同 URL） |
| URL | 自由字串 256 byte（http/https、host、port、path、query 全可自填） |
| Method | GET 或 POST |
| 認證 | 帳號＋密碼（各上限 64 字元） |
| POST body | JPEG／JSON／自訂文字（自訂字串上限 256 byte） |
| 自訂 Header | 最多 4 組（Key ≤32B／Value ≤256B） |
| 程式事件 | 最多 20 組，每組事件可觸發最多 4 個動作，並附「測試執行」鍵 |

建議用 **POST + JSON** 把事件內容塞進 body（鍵名盡量短，注意 256 byte 上限）：

```http
POST /trigger HTTP/1.1
Host: <接收端>:<port>
Content-Type: application/json

{"cam":"北門","ev":"motion","zone":3}
```

也可改用 GET 全塞在 query：`GET /trigger?cam=北門&ev=motion&zone=3`。實際欄位由接收端定義，攝影機端照填即可。

!!! note "「測試執行」鍵很好用"
    設定完成後，程式事件內附測試鍵，不必真的觸發偵測就能發一次通知，方便驗證接收端有沒有收到。

## 路線 A：透過 CX（乾接點）

!!! warning "⚠ 尚未實機驗證（依手冊／規格研究）"
    這條「攝影機接點 → CX → 廣播」的鏈路**目前只做過紙上研究、尚未實機接線測試**。導入前請先台架驗證。

!!! note "✅ 用的是原廠標準「功能」"
    本路線只用到 CX-1000 原廠標準功能（控制輸入 → 廣播事件），屬 TOA 原廠手冊涵蓋範圍、不需自研軟體。（注意：「原廠功能」≠「已實測」——功能是原廠的，但這條接線我們還沒實機跑過。）

CX-1000 的廣播啟動**只能綁在「控制輸入（乾接點）」或「站台按鍵」**，吃不到任何 HTTP／API／ONVIF 訊息。所以要讓攝影機驅動 CX 廣播，唯一有據的路是把攝影機事件**轉成乾接點**接進 CX。

流程：

```
攝影機接點輸出（開集極 DC30V／50mA）
      → 隔離繼電器
      → CX 控制輸入（無電壓 make 乾接點）
      → CX 啟動一般／緊急廣播樣式
      → 送往 IP-A1
```

!!! warning "接線注意"
    - N-C6 半球機身**不內建接點端子**，需加選配線（如 N-OP600CA）才有 1 進 1 出。
    - 攝影機接點輸出是**開集極（有極性、參考攝影機 GND、無隔離）**，接進 CX 控制輸入前**建議加一顆隔離繼電器**，避免共地與極性問題。
    - CX 控制輸入為無電壓乾接點，繼電器的 Form-C 乾接點正好對接。

此路線硬體規格明確、規劃上可行；但**尚未實機驗證**，導入前請先台架接線測試（缺件之一：半球機需 **N-OP600CA** 選配線才有接點端子）。

## 路線 B：純網路（HTTP → IP-A1）

!!! warning "⚠ 尚未實機驗證（依手冊／規格研究）"
    路線 B 目前**只做過研究、尚未實機驗證**（含 Digest 相容性、multicast 推流實作、中控軟體對接）。導入前請台架實測。

!!! tetplus "🔧 這條多屬「軟體驅動」·「中控軟體」非原廠標準"
    路線 B 需要一套**中控軟體**承接攝影機 HTTP 通知並驅動 IP-A1（推流或呼叫 API）。我們提供 **tet-tw 台灣分公司自行研發的 IP-A1 中控軟體**（**非 TOA 原廠標準產品、不在原廠手冊範圍、非原廠背書**）；你也可用自備的等效軟體。IP-A1／CX 設備本身的標準功能不受影響。

    其中「呼叫 IP-A1 Remote API」屬 ⚙️ **原廠功能但指令規格需向 TOA 書面申請**（見下方）。

IP-A1 喇叭本身內建 multicast 接收、內建音源（Pattern）與 HTTP「Remote API」監聽，因此攝影機的 HTTP 通知可以**繞過 CX、直接對喇叭**。把聲音送到 IP-A1 有三種方式：

| 方式 | 機制 | 需向 TOA 申請文件？ | 說明 |
|---|---|---|---|
| **A. 呼叫 IP-A1 Remote API** | 用 HTTP(Digest) 觸發喇叭「內建音源 Pattern 1～20」 | **需要** | 播的是喇叭內建音檔，不必自送音訊流；需 Digest 認證 |
| **B. 自行 multicast 推流** | 把 RTP 音訊推到喇叭訂閱的 Multicast 頻道（與 CX 同機制） | **不需要** | 音訊內容完全自主，需對齊 codec；若已有中控軟體在推流則幾乎零新工 |
| C. 沿用既有中控機制 | 若已有能驅動 IP-A1 的控制軟體，直接接上 | 視現況 | 最省事 |

!!! danger "方式 A 的最大卡關點：指令格式不公開"
    IP-A1 Remote API 的**確切 URL／CGI 路徑、參數、如何指定 Pattern，TOA 不在公開文件揭露**，需以書面向 TOA（子公司／代理）正式申請「IP Audio Remote API 規格書」，逐案核准。公開的 datasheet 只證明「功能存在」，不給指令字串。

    **若改走方式 B（自行 multicast 推流），則完全不需要這份文件。** 這也是最務實的純網路做法。

方式 A 的其他條件：

- 支援 Remote API 的型號：IP-A1SC15、IP-A1PG、IP-A1AF、IP-A1PC238。
- 韌體需 **3.6.0 以上**，並先在喇叭設好 Digest 認證帳密。
- **內建音源觸發僅限 LAN**，不能經 WAN。
- 內建 Pattern 若設為「Timer」重複模式，**無法**被 Remote API／事件觸發；要能被觸發請改用 **Play Count（1～10 次）或 Duration（5～3600 秒）**。

!!! tip "用中控軟體當中介，可迴避 Digest 相容疑點"
    攝影機 HTTP 通知的帳密欄未明講是 Basic 還是 Digest，直接對 IP-A1 Remote API 能否過 Digest 挑戰**尚需台架實測**。若讓攝影機改打「自建中控軟體的 webhook」、由軟體再以方式 B/C 驅動喇叭，Digest 由軟體處理，此疑點即消失。

設定路徑（IP-A1 接收端）：

```
IP-A1 網頁 → 放送 Pattern（1～20）：音源檔＋音量＋重複方式
IP-A1 網頁 → Multicast 設定（頻道 1～20）：接收 IP／埠／codec
IP-A1 網頁 → 優先權設定（Priority）：排定各來源順序
```

可選 codec：G.711u／G.711a／**G.722（實務建議）**。

## 與 CX 廣播並存：優先權怎麼排

同一顆 IP-A1 喇叭是「同時聽多個來源、以優先權仲裁」的裝置，**不是選一個模式**。所以它可以同時當「CX 廣播接收端」又被「攝影機 HTTP 直觸發」，不必二選一。撞在一起時是**單一最高優先權播出（非混音）**，較低者被丟棄，高優先權結束後若原來源還在送封包則恢復。

IP-A1 接收端的預設優先權（可自訂順序）：

```
[高]  SIP
      VMS 廣播（ONVIF）
      Multicast 1～10        ← CX 廣播常走這層
      Pattern 1～20          ← Remote API／HTTP 觸發的內建音源走這層
      Multicast 11～20
[低]  Local 廣播
```

設計原則：

- 要讓「闖入警告」**蓋過** CX 例行廣播 → 把警告的 Pattern（或你推的 multicast 頻道）排在 CX 那層**之上**。
- 反之，若 CX 廣播必須優先 → 把警告排在 CX 那層**之下**。
- 若攝影機警告與 CX 走 multicast，**務必用與 CX 不同的 multicast 頻道／位址**，避免撞頻。

!!! warning "CX 與 IP-A1 是單向"
    CX → IP-A1 只有單向送播，IP-A1 **不能反向廣播回 CX、也不能控制 CX**。CX 側不需要（也無法）去整合網路事件；你只需掌握「哪些喇叭同時也是 CX 目的地」，避開撞頻並排好優先權即可。

!!! note "兩種優先權別搞混"
    CX 系統內部有自己的一套廣播優先權（緊急高於一般高於 BGM），管的是「CX 主機要送哪一路音訊出去」；上面這張表則是「喇叭端同時收到多個網路來源時要播哪一個」。跨系統的實際壓制關係與時序，請於現場／台架實測確認。

## 實測 SOP（建議測試順序 · 尚未執行）

!!! warning "這是「待執行」的測試步驟，不是已完成的驗證"
    以下是**規劃好、但尚未實際跑過**的路線 A／B 測試順序。等實機驗證完成，本頁會把對應路線改標為 ✅ 並補上實測記錄（如上方「OP 監看」那段）。

實地測試建議「由淺入深、逐段確認」——每一段都能獨立驗證，出問題容易定位。

### STEP 0 — 前置
1. **重配攝影機 IP**：N-C6 出廠預設 IP 常與 CX-SM1000 的 `.1` 衝突，先改到現場網段相容、不撞任何設備的位址。
2. 確認攝影機與接收端（中控軟體／IP-A1／CX）在**同一網段**、可互通（ping 得到）。
3. 若走純網路對 IP-A1：確認 IP-A1 韌體 **3.6.0 以上**。

### STEP 1 — 先確認攝影機「偵測」會動
1. 攝影機網頁 → 偵測設定 → 開 **動体偵測**，畫好偵測區、調感度與反向遮罩。
2. 在鏡頭前製造動作，確認偵測有觸發（事件記錄可看）。
3. ⚠ 記得：這是動体偵測、**不分人**；要人物／滯留辨識得另裝 AI 擴充 App。

### STEP 2 — 單獨測「HTTP 通知」有沒有送出（先不接喇叭）
先用一個簡單接收端把攝影機通知「收下來看」，確認格式對、有送出：

1. 攝影機網頁 → 通知先設定 → HTTP(S) 通知：填**測試接收端** URL、Method（建議 POST + JSON）、（如需）帳密、body。
2. 攝影機網頁 → 程式事件：事件 = 動体偵測 → 動作 = 剛設的 HTTP 通知。
3. 按程式事件的 **「測試執行」** 鍵 —— 不必真的觸發偵測就會發一次。
4. 在接收端確認收到、內容正確。

    臨時測試接收端（在測試 PC 上跑，會把收到的請求印出來；攝影機 URL 指向 `http://<PC_IP>:8088/trigger`）：

    ```python
    # test_listener.py   →   python test_listener.py
    from http.server import BaseHTTPRequestHandler, HTTPServer
    class H(BaseHTTPRequestHandler):
        def _log(self):
            n = int(self.headers.get('Content-Length', 0))
            body = self.rfile.read(n).decode('utf-8', 'ignore') if n else ''
            print(self.command, self.path, '| body:', body)
            self.send_response(200); self.end_headers()
        do_GET = do_POST = _log
    HTTPServer(('0.0.0.0', 8088), H).serve_forever()
    ```

### STEP 3（路線 B）— HTTP → 中控軟體（tet-tw 自研／或自備）→ IP-A1
1. 把攝影機 HTTP 通知改指向**中控軟體的 webhook**（tet-tw 自研中控軟體，或你自備的等效軟體）。
2. 中控收到後，用 **multicast 推流**（免申請）或既有機制驅動 IP-A1 播放。
3. IP-A1 端：設好接收頻道（IP／埠／codec，建議 G.722）、Pattern、優先權。
4. 觸發偵測 → 確認目標 IP-A1 喇叭播出。

!!! tip
    直接讓攝影機打 IP-A1 的 Remote API 也行，但**指令格式需向 TOA 書面申請**、且 Digest 相容性未證實。用中控軟體當中介最穩，也迴避 Digest 疑點。

### STEP 4（路線 A）— 接點 → CX → 廣播
1. 攝影機加裝 **N-OP600CA** 選配線（半球機身才有接點端子）。
2. 攝影機接點輸出（開集極）→ **隔離繼電器** → CX-AF1062 控制輸入（無電壓乾接點）。
    - ⚠ 開集極有極性、參考攝影機 GND、無隔離 → **中間務必加隔離繼電器**，把它還原成乾接點再進 CX。
3. 攝影機程式事件：動作 = **接點輸出 ON**（模式 Pulse，1～60 秒可設）。
4. CX 設定軟體 → `Event activation → Control input`：把該 AF1062 控制輸入綁到要的廣播（一般廣播樣式，或緊急樣式）。
5. 觸發偵測 → 確認 CX 啟動廣播、送到目標喇叭。

### STEP 5 — 端到端 + 優先權
1. 完整走一次：製造事件 → 廣播播出。
2. 若同一顆 IP-A1 同時是 CX 目的地，測「攝影機警告」與「CX 廣播」**撞在一起**時誰蓋誰，依需要調 IP-A1 優先權表。
3. 記錄延遲與誤觸率，回頭調偵測感度／遮罩。

!!! warning "以現場實測為準"
    跨系統（攝影機／CX／IP-A1）的實際時序與壓制關係，務必以現場／台架實測確認，別只憑規格表推論。

## 環境與限制檢查表

| 項目 | 要點 |
|---|---|
| 攝影機偵測 | 內建僅動体＋妨害；人物／滯留辨識需 AI 擴充 App（另議） |
| 攝影機 IP | 出廠預設 IP 常與 CX-SM1000 的 `.1` 衝突，佈署時務必重配 |
| IP-A1 韌體 | Remote API 需 3.6.0 以上 |
| Remote API（方式 A） | 指令格式須向 TOA 書面申請；需 Digest；僅限 LAN；Pattern 勿用 Timer |
| Multicast（方式 B） | 免申請；codec 兩端一致（建議 G.722）；頻道／位址勿與 CX 相撞 |
| 認證相容 | 攝影機直打 IP-A1 Digest 尚待實測；走中控軟體中介可迴避 |
| 法規 | 非消防認證，不可取代法定火警／疏散系統 |

## 重點整理

- 攝影機事件可用**乾接點（經 CX）**或**HTTP（直接對 IP-A1）**兩條路觸發廣播，兩者可並存、可先後導入。（⚠ 這兩條**目前尚未實機驗證**；本頁只有「OP 監看攝影機」已實測成功。）
- N-C6 **內建偵測是動体偵測，不是 AI 人物／滯留辨識**；後者需另裝擴充 App，需求要先對清楚。
- CX 廣播啟動**只吃乾接點或按鍵**，網路事件進不了 CX，所以純網路方案要繞過 CX、直接對 IP-A1。
- 送聲音到 IP-A1 三種方式中，**Remote API 需向 TOA 申請不公開的指令規格書**；**自行 multicast 推流則免申請**，是最務實的純網路做法。
- 同一顆喇叭可同時是 CX 接收端與攝影機警告目的地，靠**優先權表**決定誰蓋誰；走 multicast 記得與 CX **分開頻道**。
- 全案**非消防認證**，僅為一般保全／導引應用；跨系統時序與優先權請以現場實測為準。
